Engagement Type: Summer Internship. The project is an outcome of Inclusive Design Institutes' participation in Google Summer of Code.

Objective

Accessibility to any digital resource (websites, user interfaces etc.) depends on the way it is served to the user. Problems in accessing can be caused due to disability, illiteracy, digital illiteracy or ageing. We must consider the user's ease of access. One way to address this problem is by using preferences. Preferences are choices of how the user wants the resource to be served. Some examples of digital preferences are font settings, contrast settings and¬†text to speech.

We start with collecting user‚Äôs digital preferences. Then we apply them across all the interfaces that the user uses. This results in experiences that are consistent, familiar and consider user's ease of access.

To understand the idea take this example. You are reading this page with black text on a white background. But, maybe you want less stress on your eyes and you prefer white text on a black background. Now, that is your preference which makes accessing this page easy for you. This also illustrates a beautiful aspect of Inclusive design, that it provides benefits to everyone, not just people with disabilities.

Thus, discovering user's preferences and applying them across digital resources is useful. This project aims to tackle the first part - developing an application that helps users to discover their preferences.

There have been attempts to address the problem of preference discovery. One implementation of this is First Discovery Tool (FDT) by Global Public Inclusive Infrastructure (GPII).

First Discovery Tool is like a wizard that takes you through different steps where you select your preferences. The ones shown here are language, text-to-speech, contrast and text-size.

Though First Discovery Tool fulfils the objective of collecting preferences. But, going through it feels like a clinical evaluation. This does not align with the principles of inclusive design. Let's take OXO potato peeler for example. The peeler allows someone with arthritis to hold a comfortable and steady grip while peeling, but it‚Äôs also ergonomic, allowing kitchen staff to avoid repetitive strain injury. This makes OXO peeler inclusive as both kitchen staff and someone with arthritis uses it in the same way. Similarly, we don't want people to feel that preference discovery is just for people with disabilities. We want to encourage everyone to discover their preference. First Discovery Tool does not feel like it's designed for everyone.

‚ùù The goal is to get the user to think deeply about his/her prefs and make selections based on those insights and metacognition. First of all, in a social sense, we want everyone to be aware of and activated about their preferences -- building the culture of usability/a11y and autonomy. it's better to create tools that go for that directly, without burying the concept in a short-term pragmatic approach (like FDT). If the medicine is already sweet, we don't have to hide it in a cupcake, and what is sweeter than narcissism ("my needs and preferences are special and unique and it's fun to learn more about myself")? We need a way to add game-like features in First Discovery for motivating and engaging users, breaking away from a 'clinical evaluation' model, and generating buzz. ‚ùû

‚Äî Jim Tobias, Inclusivity expert. Creator of FDT. From a discussion with him.

With an in-depth understanding of the problem, I was able to lay down the motives for the project.

Motives

Creating an application that does the following -

  1. Makes the process of discovering preferences fun.
  2. Helps user to learn about each preference.
  3. Allows users to play with different combinations of preferences.

Process

The project would not have been possible without the amazing mentorship of Dana Ayotte, she introduced me to the world of user experience and it's practices that were used throughout the project. The first half of the project was design-focused while the latter half was focused on development. Also, I had weekly retrospective and future planning session with the mentors.

Designing

We started by addressing the motives that we had laid out during the early phases of the project.

1.) Make the process of discovering preferences fun. We decided to create a game. This way we could deliver the idea of preference discovery excitingly and engagingly to the user. For a game to be engaging, it needs a good story.

Coming up with a storyline that incorporates preference discovery took us a lot of time. Eventually, we finalised on a storyline where the protagonist is a Cat. Conflict arises when an evil rat takes away the cat's yarn ball to his rathouse and hides it in a safe. The passcodes for the safe are spread across different rooms in the rathouse. On her quest, the cat would need to explore the house and find passcodes, so that she can get her yarn ball back.

Evil rat stealing the ball of yarn.

Passcode pieces are spread across the rooms.

2.) Help user to learn about each preference

Trying out color and high contrast configurations.

Cat has to visit these rooms to get the passcodes. Each room in the rathouse is designed to help the user learn about a particular preference and explore the different ways in which it can be configured. For example, there is a room for colour and contrast preference where users can learn about the preference and try its different configurations.

3.) Allows user to play with different combinations of preferences

Cat opens the backpack (left-bottom) and turns the screen to high contrast mode using the paint brush tool. You can see the different tools that the cat has collected in the backpack.

While exploring each room, the cat gets a tool that can be used to change the configuration of the preference. For example, in the color and contrast room, the cat gets a paint brush, which can be used to switch the display between color and high contrast mode. Similarly, the cat finds tools while visiting the rooms and keeps them in her backpack. Once a tool is collected it can be used at any time. This way cat can try out different combinations of preferences.

We finalised the game design and storyline over multiple iterations. I created low-fidelity mockups (sketches) for the game every week and we had discussions over them. These sketches helped us to quickly discuss different aspects of the game. You can check out all the sketches we created before finalising story, characters, room designs, and other game elements.

Continuous Feedback

From the very first week of development, I made my friends and family play the game to get feedback. I used to observe them while playing the game to identify the parts of the game that were hard, boring or unintuitive.

For example - During early versions of the game, when the cat became interactable for the first time. Users did not understand that they were required to take an action. I observed my friends struggling at this point. So, I added arrow and enter keys on screen to tell users about the possible actions.

In the top-left side of the screen you can see arrow keys and enter signifying possible actions.

Creating game assets

From my early college days, I have been working with illustrations, posters, logos and other forms of digital media. Expertise in visual communication turned out to be immensely helpful while designing the characters, backgrounds, and other items in the game. Also, my experience with tools like Adobe illustrator helped me to convert the low-fidelity sketches into high-quality assets for the game.

Sound Design for the game

I used a free Digital Audio Workstation to create the music. Knowing the basics of music theory helped me with the task. Major Scales (key of G and C were used) have a positive and optimistic feel to them, I wanted the same feeling while players are on the quest to get the yarn balls back. On the other hand, minor scales have a sad or emotional feeling associated with them. Evil rat taking away the yarn ball is a situation where I used E minor scale.

Bootstrapping the project

The project repository is hosted on¬†Github. Phaser and Infusion Frameworks

Phaser is a game engine, and Infusion is a JavaScript framework for creating applications that are highly accessible and reusable. Challenge was using them together for the project. To understand how these two work together in Discovery Cat, let us get an understanding of the structure of Phaser and Infusion -

Every game in Phaser has a parent game object around which the game is built. It is created using Phaser.game. Another important thing in Phaser are States. States are scenes in a game. Game levels, loading screens and main menu are all examples of states. States are the building blocks which are added to the parent game object to make the game. Each state is a constructor function with prototype methods create, preload and update attached to it. These methods are used to define the logic and functionality of the scene. You can read more about them in the¬†Phaser tutorial.

Infusion on the other hand is a framework made with the intent of reusability in mind. The implementation unit of Infusion is called a component. You can learn more Infusion¬†here.

I created a wrapper for Phaser using Infusion, it enabled me to use Phaser through Infusion. To achieve this I created a Infusion component which when instantiated, creates a Phaser.game object and stores it. Next, I created each state for the game as a component. I added these components as sub-components to the main component containing the Phaser game object. This way the game states (sub-components) can access the game object and the game object can also access these states. This is the basic ideology behind the code of Discovery Cat and might help you to understand the code easily.

Learning about web-accessibility: WebSpeech API

Web speech API enables browsers to handle voice data. It has two parts - one is voice recognition and other one is voice synthesis. For the project we particularly required speech synthesis for adding the feature of text to speech(Screen Reader) to the game. The need to use it rose as the HTML5 game engines like Phaser uses canvas and normal screen reader cannot read the canvas. So I chose web speech API for text to speech support. If you want to know how I solved the issue you can find it in my blog on midterm evaluation.

During the course of this project I answered my first Stack Overflow answer and learned about cool things like colorful console logs. You can read about them in the project blog.

Conclusion

The game was well received by everyone at the Inclusive Design Institute during the final presentation. Recently, IDI made the game live and it is probably being used by people. Though, I am not aware of the feedback it has been getting.
